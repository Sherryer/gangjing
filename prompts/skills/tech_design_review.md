# 技术方案/架构设计审查 Skill
# 适用类型：tech_design / architecture / system_design / migration_plan
# 版本：v1.0

## 额外审查维度（在通用检查清单之上）

| 维度 | 图标 | 关注点 |
|-----|------|-------|
| 架构合理性 | 🔴 | 组件划分是否清晰、职责是否单一、是否过度设计或设计不足、分层是否合理 |
| 可扩展性 | 🟡 | 流量增长10x/100x时架构是否扛得住、是否有水平扩展能力、瓶颈在哪 |
| 容灾与高可用 | 🟣 | 单点故障识别、故障切换机制、数据一致性保障、RTO/RPO是否明确 |
| 技术选型 | 🔵 | 选型理由是否充分、是否有对比评估、社区活跃度/团队熟悉度是否考虑 |
| 接口与协议 | 🟢 | API设计是否RESTful/合理、协议版本兼容性、上下游依赖是否明确 |
| 安全设计 | 🔴 | 认证鉴权方案、数据加密传输/存储、敏感信息保护、攻击面分析 |
| 可运维性 | 🟡 | 监控告警覆盖、日志规范、灰度发布/回滚方案、配置管理 |

## P 级标准（技术方案专用）

- P0 🔴：架构存在单点故障 / 数据可能丢失 / 有安全漏洞 / 核心链路无容灾
- P1 🟠：技术选型有明显更优方案 / 扩展性瓶颈未解决 / 接口设计有兼容性风险
- P2 🟡：方案可行但不够优雅 / 缺少监控告警设计 / 文档描述不够清晰
- P3 🔵：命名规范 / 图表可读性 / 补充性能基准数据 / 格式优化

## 技术方案审查专用规则

1. 对每个架构决策追问"为什么选这个方案？对比过哪些替代方案？"
2. 画出关键链路的故障传播路径：A挂了→影响B→导致C不可用，检查是否有兜底
3. 检查容量评估：QPS/TPS预估是否合理、存储增长是否有规划、带宽是否够用
4. 检查数据流完整性：数据从产生到消费的全链路是否有丢失/重复/乱序风险
5. 检查向前兼容性：新方案上线时老系统/老数据如何平滑迁移
6. 检查依赖合理性：是否引入了不必要的外部依赖、依赖挂了有无降级方案
7. 对"未来再优化"的说法保持警惕：如果是P0级问题，不接受延后处理
