# 代码审查 Skill
# 适用类型：code / pull_request / code_snippet
# 版本：v1.0

## 额外审查维度（在通用检查清单之上）

| 维度 | 图标 | 关注点 |
|-----|------|-------|
| 正确性 | 🔴 | 逻辑错误、边界 case、空指针/undefined、类型不安全、竞态条件 |
| 性能 | 🟡 | 不必要的重渲染/重计算、内存泄漏、O(n²) 复杂度、大列表未虚拟化 |
| 安全性 | 🟣 | XSS、SQL注入、敏感信息硬编码、不安全的反序列化、CSRF |
| 可维护性 | 🔵 | 命名模糊、过度耦合、违反单一职责、魔法数字、缺少类型定义 |
| 工程规范 | 🟢 | 错误处理缺失、日志不足、测试覆盖不足、文档缺失 |

## P 级标准（代码专用，覆盖通用标准）

- P0 🔴：线上会崩 / 数据会丢 / 有安全漏洞
- P1 🟠：功能不符合预期 / 严重性能问题 / 会影响其他模块
- P2 🟡：能跑但不好维护 / 有更优方案 / 轻度性能问题
- P3 🔵：命名 / 风格 / 注释 / 微小优化

## 代码审查专用规则

1. 精确到行号定位问题（如："第 23 行" 或 "fetchData 函数第 5 行"）
2. 每个 P0/P1 问题必须给出修复后的代码片段（不只是描述）
3. 关注框架特有的最佳实践（React hooks 规则、Vue 响应式陷阱、async/await 错误处理等）
4. 检查 TypeScript 类型是否足够严格（any 类型要标记）
5. 检查异步操作的错误处理和竞态条件
6. 检查组件/资源的卸载清理（useEffect cleanup、事件解绑、定时器清除等）
