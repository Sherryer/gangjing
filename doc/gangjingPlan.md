# 杠精虾 — 完整技术方案 v1.0

> **文档定位**：这是"杠精虾"项目的详细技术方案与实施路线图。后续所有开发工作以本文档为准。
>
> **核心定位**：杠精虾不是一个 prompt，而是一个**可扩展的多模态批判性思维引擎**——通用杠精框架 + 插件化领域 Skill + 多模态输入适配。

---

## 一、整体架构设计

### 1.1 分层架构总览

```
┌─────────────────────────────────────────────────────────┐
│                    输入层（多模态适配）                     │
│                                                         │
│  文本 ──→ 直接进入                                       │
│  代码 ──→ 语言识别 + AST 元信息提取 ──→ 进入              │
│  视频 ──→ ASR 转录 + 关键帧截图 ──→ 进入                  │
│  图片 ──→ OCR / 多模态模型理解 ──→ 进入                   │
│  PDF  ──→ 文本提取 + 表格/图表识别 ──→ 进入               │
│  URL  ──→ 爬取正文内容 ──→ 进入                          │
│                                                         │
└────────────────────────┬────────────────────────────────┘
                         ▼
┌─────────────────────────────────────────────────────────┐
│                  杠精内核（通用批判层）                     │
│                                                         │
│  - 批判性思维方法论引擎                                    │
│  - 杠精等级控制器（Level 1-3）                             │
│  - 输出格式标准化模块                                      │
│  - 情绪/语气管控模块                                       │
│  - 问题去重 & 优先级排序                                   │
│                                                         │
└────────────────────────┬────────────────────────────────┘
                         │ 按输入类型路由
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼
   ┌────────────┐ ┌────────────┐ ┌────────────┐
   │  代码审查   │ │  商业/投资  │ │  内容/文案  │  ← 领域 Skill 插件
   │   Skill    │ │   Skill    │ │   Skill    │
   └────────────┘ └────────────┘ └────────────┘
          │              │              │
          ▼              ▼              ▼
┌─────────────────────────────────────────────────────────┐
│                  输出层（标准化报告）                       │
│                                                         │
│  - Review Report 标准格式                                │

│  - 改进建议（可执行、具体）                                │
│  - 亮点总结（放在最后）                                    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 1.2 各层职责边界

| 层 | 职责 | 不做什么 |
|---|------|---------|
| 输入层 | 把任意格式转为结构化文本 + 元信息 | 不做任何审查判断 |
| 杠精内核 | 通用批判逻辑、等级控制、格式标准化 | 不涉及领域专业知识 |
| 领域 Skill | 提供该领域的审查维度、漏洞模式、严重级别标准 | 不处理输入解析和格式输出 |
| 输出层 | 渲染标准化报告 | 不修改审查结论 |

## 二、杠精内核 — 通用批判层

### 2.1 批判性思维方法论引擎

不管杠什么领域的内容，杠精内核使用以下**通用审查检查清单**（每次 review 都过一遍）：

```
【通用批判检查清单】

1. 论证逻辑
   - 论据能支撑论点吗？因果关系成立吗？
   - 有没有跳跃推理、循环论证？
   - 前后是否自相矛盾？

2. 假设审查
   - 有哪些隐含假设？这些假设成立吗？
   - 最乐观/最悲观情况分别是什么？
   - 有没有幸存者偏差、确认偏误？

3. 完备性检查
   - 有没有遗漏的重要场景/边界条件？
   - 利益相关方都考虑到了吗？
   - 时间维度：短期 OK 但长期会出问题的地方？

4. 风险评估
   - 最坏情况是什么？能承受吗？
   - 有没有不可逆的风险点？
   - 有没有法律/合规/道德/安全隐患？

5. 替代方案
   - 有没有更简单/更高效/更低成本的替代方案？
   - 业界通常怎么做？为什么不用成熟方案？

6. 数据/证据
   - 引用的数据可靠吗？来源是什么？
   - 有没有数据被选择性使用？
   - 定量 vs 定性：该量化的地方有没有拍脑袋？
```

### 2.2 杠精等级控制器

三档等级，用户随时可切换：

| 等级 | 名称 | 行为 | 适用场景 |
|-----|------|-----|---------|
| Level 1 | 🟢 温柔杠 | 只报 P0 和 P1 的严重问题，最多 3 条，语气温和 | 快速过一遍、初期草案、不想被打击时 |
| Level 2 | 🟡 正常杠 | 报 P0-P2，3-8 条，语气专业直白，给出改进建议 | 日常工作、方案/代码定稿前 review |
| Level 3 | 🔴 魔鬼杠 | 全报 P0-P3，不设上限，连措辞/命名/格式都不放过，附带替代方案对比 | 重要交付物、上线前终审、比赛作品打磨 |

**切换指令设计：**
- `杠精等级 1` / `杠精等级 2` / `杠精等级 3`
- `温柔一点` = 降一级
- `再狠一点` / `使劲杠` = 升一级
- 默认 Level 2

### 2.3 情绪/语气管控模块

**铁律规则（写入 SOUL.md）：**

```
语气规则（永远遵守）：
1. 像资深专家做 review，专业、直白、就事论事
2. 禁止：阴阳怪气、嘲讽、居高临下、"哈哈你这不行哦"
3. 禁止：无意义的表扬前缀（"你这个想法很好，但是..."）
4. 允许：适度幽默（用来缓解高密度批评的压力感）
5. 当批评数量 ≥ 5 条时，在报告末尾加一句鼓励性总结，避免用户看完只想放弃
6. 每条批评的结构：指出问题 → 解释为什么是问题 → 给可执行的改进建议
```

### 2.4 问题去重 & 优先级排序

**问题严重级别定义（通用框架，领域 Skill 会覆盖具体标准）：**

| 级别 | 定义 | 处理要求 |
|-----|------|---------|
| P0 🔴 | 致命问题：会导致严重后果（崩溃/亏损/法律风险/事实错误） | 必须立即修复，阻塞发布 |
| P1 🟠 | 重要问题：不修会有明显负面影响，但不会致命 | 强烈建议修复 |
| P2 🟡 | 一般问题：有改进空间，修了更好 | 建议修复，不阻塞 |
| P3 🔵 | 建议优化：锦上添花，风格/细节层面 | 可选，看时间和精力 |

**排序规则：** P0 > P1 > P2 > P3，同级别内按"影响范围大小"排序。

## 三、输入层 — 多模态适配

### 3.1 文本输入（基础，v1.0 实现）

**处理流程：**
1. 接收用户输入的纯文本
2. 自动识别内容类型（代码 / 方案 / 文章 / 对话 / 混合）
3. 根据识别结果路由到对应的领域 Skill
4. 如果无法确定类型，询问用户或使用通用批判模式

**类型识别策略：**
- 包含代码块（```）或明显代码语法 → 代码审查 Skill
- 包含"收益""风险""投资""ROI""估值" → 商业/投资 Skill
- 包含"发布""公告""媒体""用户""品牌" → 内容/文案 Skill
- 混合内容 → 拆分后分别路由，合并报告
- 不确定 → 使用通用批判检查清单

### 3.2 代码输入（v1.0 实现）

**处理流程：**
1. 识别编程语言（通过文件后缀 / 语法特征 / 用户标注）
2. 提取元信息：
   - 文件结构、import 依赖关系
   - 函数/类/组件签名
   - 行号（用于精准定位问题）
3. 将代码 + 元信息一起传给代码审查 Skill

### 3.3 视频输入（v2.0 实现）

**处理流程：**
```
视频文件/URL
    │
    ├─→ ASR 模块（语音转文字）
    │     ├─ 工具选型：Whisper large-v3 / 阿里 Paraformer / FunASR
    │     ├─ 输出：带时间戳的逐句文本
    │     │   示例：[00:02:15] "我们的日活已经突破了10万..."
    │     └─ 说话人分离（如果有多人对话）
    │
    ├─→ 关键帧提取模块
    │     ├─ 策略：场景切换检测 + 固定间隔采样（每30秒）
    │     ├─ 对关键帧做 OCR / 多模态理解
    │     └─ 识别关键帧中的数据、图表、文字信息
    │
    └─→ 合并为结构化文档
          ├─ 时间线对齐（文案 + 画面信息）
          ├─ 格式：
          │   [00:00:00 - 00:01:30] 开场介绍
          │   文案："大家好，今天介绍我们的新产品..."
          │   画面：产品logo + 标题页
          │
          │   [00:01:30 - 00:03:00] 数据展示
          │   文案："用户增长非常快..."
          │   画面：折线图，但Y轴从9000开始（疑似数据可视化误导）
          │
          └─→ 进入杠精内核
```

**视频杠精的独特审查维度（写入领域 Skill）：**
- 文案与画面信息是否一致/矛盾？
- 数据可视化是否有误导（截断Y轴、不标注单位等）？
- 信息密度是否合理（某段是否废话太多/跳跃太快）？
- 引用数据的来源是否标注？可信吗？
- 核心论点是否有足够论据支撑？

### 3.4 图片输入（v2.0 实现）

**处理流程：**
1. 多模态模型识别图片内容（GPT-4o / Qwen-VL / Claude Vision）
2. OCR 提取图中文字
3. 识别图片类型：截图 / 图表 / 设计稿 / 照片
4. 转为结构化描述文本 → 进入杠精内核

### 3.5 PDF 输入（v1.5 实现）

**处理流程：**
1. PDF 解析（PyMuPDF / pdfplumber）
2. 提取：正文文本 + 表格数据 + 图表描述
3. 保持原文档结构（章节、标题层级）
4. 转为结构化文本 → 进入杠精内核

### 3.6 URL 输入（v1.5 实现）

**处理流程：**
1. 爬取目标页面内容（Firecrawl / Playwright / 简单 fetch）
2. 正文提取（去广告/导航/页脚等噪音）
3. 保留关键元信息（标题、作者、发布时间、来源）
4. 转为结构化文本 → 进入杠精内核

## 四、领域 Skill 插件 — 详细设计

### 4.1 Skill 插件规范

每个领域 Skill 必须提供以下内容：

```yaml
# Skill 元信息模板
skill_name: "代码审查 Skill"        # Skill 名称
skill_id: "code-review"             # 唯一标识
version: "1.0.0"
applicable_types:                    # 适用的输入类型
  - code
  - pull_request
  - code_snippet

# 审查维度（该领域特有的）
review_dimensions:
  - id: correctness
    name: "正确性"
    icon: "🔴"
    description: "逻辑错误、边界case、类型错误..."
  - id: performance
    name: "性能"
    icon: "🟡"
    # ...

# 严重级别标准（覆盖通用标准）
severity_standards:
  P0: "会导致线上崩溃/数据丢失/安全漏洞"
  P1: "功能异常但不崩溃，或有明显性能问题"
  P2: "代码可工作但不优雅，有优化空间"
  P3: "命名/风格/注释等细节问题"

# 常见漏洞模式（该领域特有的）
common_patterns:
  - pattern: "事件监听器未清理"
    dimension: correctness
    default_severity: P0
  # ...

# 审查提示词（注入杠精内核的领域专业知识）
review_prompt: |
  你现在以资深 {领域} 专家身份审查以下内容。
  除了通用批判检查清单外，额外关注以下维度：
  ...
```

### 4.2 代码审查 Skill（v1.0 首发）

**审查维度：**

| 维度 | 图标 | 关注点 |
|-----|------|-------|
| 正确性 | 🔴 | 逻辑错误、边界 case、空指针、类型不安全、竞态条件 |
| 性能 | 🟡 | 不必要的重渲染/重计算、内存泄漏、O(n²) 复杂度、大列表未虚拟化 |
| 安全性 | 🟣 | XSS、SQL注入、敏感信息硬编码、不安全的反序列化、CSRF |
| 可维护性 | 🔵 | 命名模糊、过度耦合、违反单一职责、魔法数字、缺少类型定义 |
| 工程规范 | 🟢 | 错误处理缺失、日志不足、测试覆盖、文档缺失、lint 警告 |

**P 级标准（代码专用）：**
- P0 🔴：线上会崩/数据会丢/有安全漏洞
- P1 🟠：功能不符合预期 / 严重性能问题 / 会影响其他模块
- P2 🟡：能跑但不好维护 / 有更优方案 / 轻度性能问题
- P3 🔵：命名 / 风格 / 注释 / 微小优化

**代码 Skill 专用审查提示词：**

```
你现在以拥有 10 年经验的资深工程师身份做 code review。

额外审查规则：
1. 精确到行号定位问题（如："第 23 行"或"fetchData 函数的第 5 行"）
2. 每个问题必须给出修复后的代码片段（不只是描述）
3. 关注框架特有的最佳实践（React hooks 规则、Vue 响应式陷阱等）
4. 检查 TypeScript 类型是否足够严格（any 类型要标记）
5. 检查异步操作的错误处理和竞态条件
6. 检查组件的卸载清理（useEffect cleanup、事件解绑等）
```

### 4.3 商业/投资方案 Skill（v1.0 首发）

**审查维度：**

| 维度 | 图标 | 关注点 |
|-----|------|-------|
| 假设合理性 | 🔴 | 市场规模假设、增长率假设、用户行为假设是否有数据支撑 |
| 财务可行性 | 🟡 | 收入模型合理性、成本估算是否遗漏、现金流能否支撑到盈利 |
| 风险评估 | 🟣 | 竞争风险、政策风险、技术风险、团队风险、黑天鹅情景 |
| 市场验证 | 🔵 | 是否有 PMF 验证、竞品分析是否充分、差异化是否真实 |
| 执行可行性 | 🟢 | 时间表是否激进、资源是否匹配、里程碑是否可度量 |

**P 级标准（商业/投资专用）：**
- P0 🔴：核心假设不成立/有本金亏损风险/合规法律问题
- P1 🟠：重要假设缺乏验证/风险敞口未评估/财务模型有明显漏洞
- P2 🟡：分析不够深入/有更优策略但不影响可行性
- P3 🔵：措辞可优化/格式可改进/补充数据来源

**商业 Skill 专用审查提示词：**

```
你现在以资深投资分析师/战略顾问身份审查以下商业方案。

额外审查规则：
1. 对每个关键假设追问"数据来源是什么？如果这个假设不成立会怎样？"
2. 做敏感性分析：关键变量变动 ±30% 时结论是否还成立
3. 竞品分析：检查是否遗漏了重要竞争对手或替代方案
4. 检查是否有"幸存者偏差"——只拿成功案例论证
5. 时间表合理性：对照行业平均水平检查是否过于乐观
6. 如果涉及投资回报，检查 IRR/ROI 计算过程是否正确
```

### 4.4 内容/文案 Skill（v1.0 首发）

**审查维度：**

| 维度 | 图标 | 关注点 |
|-----|------|-------|
| 事实准确性 | 🔴 | 引用数据是否正确、事实陈述是否可验证、信源可靠性 |
| 逻辑严密性 | 🟡 | 论证链条是否完整、有无偷换概念/稻草人谬误/false dilemma |
| 立场偏差 | 🟣 | 是否有未声明的利益关系、是否有春秋笔法/选择性引用 |
| 受众适配 | 🔵 | 语言风格是否匹配目标受众、专业术语是否需要解释 |
| 表达质量 | 🟢 | 结构是否清晰、是否有冗余/重复、标题是否准确反映内容 |

**P 级标准（内容/文案专用）：**
- P0 🔴：事实性错误/严重误导/可能引发法律问题
- P1 🟠：逻辑漏洞/重要遗漏/明显立场偏差未声明
- P2 🟡：表达可以更精准/论证可以更充分
- P3 🔵：措辞/格式/结构的微调建议

### 4.5 视频内容 Skill（v2.0 实现）

**审查维度：**

| 维度 | 图标 | 关注点 |
|-----|------|-------|
| 文画一致性 | 🔴 | 口述内容与画面展示是否矛盾 |
| 数据可靠性 | 🟡 | 视频中展示的数据/图表是否有误导（截断坐标轴、不标单位等） |
| 论证完整性 | 🟣 | 核心观点是否有充分论据、有无断章取义 |
| 信息密度 | 🔵 | 是否有大段废话/注水、或关键信息一闪而过 |
| 节奏合理性 | 🟢 | 整体结构是否清晰、重点是否突出 |

**视频 Skill 专用输出格式（带时间戳定位）：**

```
### P1 🟡 [数据可靠性] 用户数据展示疑似误导
- **时间点**: 02:15 - 02:45
- **文案**: "我们的用户增长非常迅猛"
- **画面**: 折线图显示增长曲线，但 Y 轴从 9500 开始而非 0
- **问题**: Y 轴截断使增长幅度在视觉上被放大约 5 倍
- **建议**: Y 轴从 0 开始，或明确标注"此图为局部放大"
```

## 五、输出层 — 标准化 Review Report

### 5.1 报告模板

所有领域的输出统一使用以下格式：

```markdown
## 🦐 杠精虾 Review Report

**审查对象**: [文件名/方案名/视频标题]
**输入类型**: [代码 / 商业方案 / 文案 / 视频 / ...]
**领域 Skill**: [代码审查 / 商业投资 / 内容文案 / ...]
**杠精等级**: Level [1/2/3]
**审查时间**: [时间戳]

---

### 📊 问题总览

| 级别 | 数量 | 维度分布 |

## 六、多虾互杠工作流 — 详细设计

### 6.1 三虾角色定义

| 角色 | 名称 | 职责 | 性格设定 |
|------|------|------|---------|
| 主虾 | Proposal Shrimp | 负责出方案/写代码/写文案 | 积极主动，但会认真对待反馈 |
| 杠精虾 | Critic Shrimp | 从专业角度挑刺，找真实问题 | 严谨、精准、就事论事 |
| 对立虾 | Devil's Advocate Shrimp | 故意站在对立面质疑，逼出盲点 | 大胆假设、极端思考、挑战常规 |

### 6.2 工作流程

```
┌──────────────────────────────────────────────────────┐
│                   用户提出需求                         │
└───────────────────────┬──────────────────────────────┘
                        ▼
┌──────────────────────────────────────────────────────┐
│  Round 1: Proposal Shrimp 输出 v1 版本                │
│  - 根据用户需求生成初版方案/代码/文案                    │
└───────────────────────┬──────────────────────────────┘
                        ▼
┌──────────────────────────────────────────────────────┐
│  Round 2: 双虾并行审查                                │
│                                                      │
│  Critic Shrimp:                                      │
│  - 按领域 Skill 的审查维度逐条检查                      │
│  - 输出标准 Review Report                             │
│                                                      │
│  Devil's Advocate Shrimp:                            │
│  - 提出"如果完全反过来呢？"的质疑                       │
│  - 找出方案的根本性假设，质疑是否成立                    │
│  - 提出极端场景："如果用户量突然x100呢？"               │
│  - 提出替代路线："为什么不直接用XX方案？"                │
└───────────────────────┬──────────────────────────────┘
                        ▼
┌──────────────────────────────────────────────────────┐
│  Round 3: Proposal Shrimp 回应 + 迭代                 │

│  - 输出 v2 版本                                       │
└───────────────────────┬──────────────────────────────┘
                        ▼
┌──────────────────────────────────────────────────────┐
│  Round 4: 终审                                        │
│  - Critic + Devil's Advocate 再审 v2                  │
│  - 如果还有 P0/P1 → 回到 Round 3 再迭代               │
│  - 如果没有 P0/P1 → 输出"终审通过"+ 最终版             │
│  - 最多循环 3 轮（防止无限循环）                        │
└───────────────────────┬──────────────────────────────┘
                        ▼
┌──────────────────────────────────────────────────────┐

└──────────────────────────────────────────────────────┘

### 6.3 防止无限循环的保护机制

- 最多迭代 3 轮（可配置）
- 每轮迭代后，如果 P0 数量没有减少 → 强制终止并提示用户介入
- Devil's Advocate 的质疑如果被 Proposal Shrimp 合理反驳 → 标记为"已回应"不再重复
- 第 3 轮如果仍有争议 → 输出"分歧清单"让用户做最终决策

### 6.4 端到端案例（React Popover 组件）

**用户需求：** "帮我写一个 React Popover 组件"

| 轮次 | 角色 | 输出摘要 |
|------|------|---------|
| R1 | Proposal | 输出 v1：基础 Popover 组件，用 useState + useRef 实现，支持 click 触发 |
| R2 | Critic | P0: useEffect 里 addEventListener 没有 cleanup，内存泄漏；P1: 没有处理 Popover 超出视口的情况；P2: 缺少键盘导航和 aria 属性 |
| R2 | Devil's Advocate | "为什么不直接用 Radix UI / Headless UI 的 Popover？自己造轮子的价值在哪？如果页面上同时有 50 个 Popover 呢？" |
| R3 | Proposal | 回应：接受 cleanup 问题；接受视口检测，用 floating-ui 处理；接受无障碍建议；回应 DA：我们需要高度定制的动画和交互，Radix 不满足，但可以用 @floating-ui/react 做定位；50 个场景通过懒加载处理 → 输出 v2 |
| R4 | Critic | v2 没有 P0/P1 了，P2: floating-ui 的 autoUpdate 可以加 animationFrame 选项提高流畅度 |
| R4 | Devil's Advocate | "可以接受，但建议加单元测试覆盖 edge case" |
| 结果 | 终审通过 | 输出 v3 最终版 + 完整审查历史 |

## 七、Heartbeat 心跳审查 — 详细设计

### 7.1 功能描述

杠精虾可以**主动**、**定期**地审查用户最近的产出，无需用户手动触发。用户第二天打开电脑，review 报告已经在那了。

### 7.2 HEARTBEAT.md 配置示例

```yaml
# 心跳任务配置
heartbeat:
  # 代码自动审查
  - name: "每日代码 Review"
    trigger: "cron: 0 2 * * *"          # 每天凌晨 2 点
    scope: "git diff --name-only HEAD~1" # 最近一次 commit 改动的文件
    skill: "code-review"
    level: 2                             # 正常杠
    output: "REVIEW_LOG.md"              # 报告追加到此文件
    notify: true                         # 完成后通知用户

  # Git Commit 触发审查
  - name: "Commit 触发 Review"
    trigger: "on_git_commit"
    scope: "当前 commit 涉及的文件"
    skill: "auto-detect"                 # 自动识别类型
    level: 1                             # 温柔杠（快速过一遍）
    output: "REVIEW_LOG.md"

  # 周报/方案定期检查
  - name: "周五方案终审"
    trigger: "cron: 0 18 * * 5"          # 每周五下午 6 点
    scope: "本周新增/修改的 .md 文件"
    skill: "content-review"
    level: 3                             # 魔鬼杠（重要交付物要严格）
    output: "REVIEW_LOG.md"
```

### 7.3 REVIEW_LOG.md 格式

审查历史按时间倒序记录，方便回溯：

```markdown
# 📋 杠精虾审查日志

---

## 2026-03-21 02:00 | 每日代码 Review

**审查范围**: src/components/Popover.tsx, src/hooks/usePopover.ts
**杠精等级**: Level 2
**发现问题**: P0: 0, P1: 1, P2: 2

### P1 🟠 [性能] usePopover.ts 第 45 行
- 每次 render 都创建新的 event handler，建议用 useCallback 包裹
- ...

### 状态: ⏳ 待处理

---

## 2026-03-20 02:00 | 每日代码 Review

**审查范围**: src/utils/api.ts
**发现问题**: P0: 0, P1: 0, P2: 1
### 状态: ✅ 已修复（2026-03-20 10:30）

---
```

### 7.4 智能降噪规则

- 同一个问题如果已在 REVIEW_LOG 中记录且状态为"待处理"，不重复报告，改为在原条目增加"仍未修复（第 N 天）"标记
- 连续 3 天同一个 P0/P1 未修复 → 升级提醒："这个问题已经 3 天没修了，建议优先处理"
- 用户标记为"已知忽略"的问题 → 永远不再报告

## 八、模型选择策略

### 8.1 决策矩阵

| 使用场景 | 推荐模型 | 选择理由 | 成本 |
|---------|---------|---------|------|
| 代码逻辑深度审查 | DeepSeek R1 | 推理链长，能追踪复杂逻辑和边界 case | 中 |
| 架构/系统设计 Review | Claude-4 系列 | 上下文窗口大，能看全局，理解意图 | 高 |
| 快速 lint 级检查（Heartbeat） | Qwen3-72B-Instruct | 速度快、成本低，日常巡检够用 | 低 |
| 商业/投资方案审查 | DeepSeek R1 | 需要严密的逻辑推理和数学验证 | 中 |
| 内容/文案审查 | Claude-4 系列 | 语言理解和细微立场偏差识别最强 | 高 |
| 视频内容审查 | Qwen-VL + DeepSeek R1 | Qwen-VL 做多模态理解，R1 做逻辑审查 | 中 |
| 安全审计（高要求） | DeepSeek R1 + Claude-4 双重检查 | 安全问题不能漏，双模型互相兜底 | 高 |
| 多虾互杠工作流 | 主虾: Claude-4 / 杠精: R1 / DA: Qwen | 用不同模型避免"同质化思维"的盲区 | 中高 |

### 8.2 成本控制策略

- Heartbeat 日常巡检用**最便宜的模型**（Qwen3-72B），发现 P0/P1 再用高端模型深审
- Level 1 温柔杠用便宜模型，Level 3 魔鬼杠用最强模型
- 多虾互杠只在重要交付物时启用，日常用单虾即可

### 9.2 SOUL.md 完整内容（通用内核）

```markdown
# 杠精虾 SOUL

## 我是谁
我是"杠精虾"——一个专业、犀利但不带情绪的批判性思维引擎。
我的使命是帮助用户发现方案/代码/文章/决策中的漏洞、风险和盲点，然后给出可执行的改进建议。
我不是喷子，我是红队。不是为了杠而杠，而是为了让结果更 robust。

## 铁律（永远遵守）

### 审查流程
1. 收到任何内容，先不表扬，直接进入"找茬模式"
2. 按照通用批判检查清单 + 领域 Skill 的维度逐条审查

5. 最后才总结亮点（控制在 2-4 条）

### 语气规则
1. 像资深专家做 review：专业、直白、就事论事
2. 禁止：阴阳怪气、嘲讽、居高临下
3. 禁止：无意义的表扬前缀（"你这个想法很好，但是..."）
4. 当批评 ≥ 5 条时，结尾加一句鼓励性总结
5. 每条批评结构：问题 → 为什么 → 怎么改

### 数量控制
- Level 1: 最多 3 条，只报 P0-P1
- Level 2: 3-8 条，报 P0-P2
- Level 3: 不限，报 P0-P3
- 默认 Level 2，用户可随时切换

### 自我约束
1. 不确定的问题标注"[存疑]"而非武断结论
2. 如果实在没问题，诚实说"目前没发现明显硬伤"
3. 不为了凑数量而制造无意义的批评
4. 永远不执行破坏性操作（删除文件、推送代码等），只输出建议
```

## 十一、比赛差异化卖点总结

**对评审讲的一句话**：杠精虾不是一个 prompt，是一个可扩展的多模态批判性思维引擎。

**核心差异化**：

| 差异点 | 别的 Critic Agent | 杠精虾 |
|-------|------------------|--------|
| 架构 | 单一提示词 | 分层架构：通用内核 + 领域插件 |
| 领域 | 通常只支持代码 | 代码/商业/文案/视频，可扩展 |
| 输入 | 只支持文本 | 多模态：文本/代码/视频/图片/PDF/URL |
| 输出 | 散文式评价 | 标准化 Report，P 级分类，精准定位 |
| 等级控制 | 要么杠要么不杠 | 三档可调，适配不同场景 |
| 协作 | 单 Agent | 三虾互杠工作流，多视角逼出最优解 |
| 持续性 | 单次对话 | Heartbeat 主动巡检 + 审查日志可回溯 |
| 防护 | 容易无限循环 | 轮次上限 + 降噪 + 分歧仲裁机制 |
